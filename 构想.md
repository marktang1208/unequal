# 不等号 / unequal —— 微信端个人育儿智能体

## 一、背景

构建一个部署在微信生态中的"一问一答式"育儿智能体：

- **中文名**：不等号
- **英文名**：unequal
- **核心目标**：基于"个人育儿知识库"回答育儿问题，回答必须标注引用来源，避免 LLM 幻觉。

## 二、原始需求

1. 建立个人育儿知识库
2. 知识库来源 ①：上传 PDF / Word / TXT / Markdown 文件
3. 知识库来源 ②：抓取小红书指定账号的内容，自动入库
4. 知识库来源 ③：抓取微信公众号指定账号的内容，自动入库
5. 知识库来源 ④：抓取指定网页的内容，自动入库
6. 返回内容需标注引用来源，保证可追溯，而非大模型幻觉

## 三、技术栈选型（已确认）

| 层 | 选型 | 理由 |
|---|---|---|
| 入口 | 微信小程序（个人主体） | 富交互可展示引用卡片；无 5 秒被动回复超时；省运维 |
| 后端 | Cloudflare Workers | 免费额度够用；边缘部署；HTTPS 证书自动签发 |
| 会话状态 | Cloudflare Durable Objects | 天然适合"一个用户一段会话" |
| 向量检索 | Cloudflare Vectorize | 托管向量库，免费 3000 万维度/月 |
| 元数据 | Cloudflare D1（SQLite） | 存文档元数据、来源、用户表 |
| 原文存储 | Cloudflare R2 | 存 PDF / 原文 / 缩略图，免费 10GB |
| LLM | **MiniMax API（年卡已购）** | 中文育儿问答质量够用，年卡覆盖所有调用 |
| Embedding | **MiniMax embedding（年卡覆盖）** | 与 chat 同一供应商，pipeline 一致 |
| 管理后台 | React + Vite（独立 Web） | 源管理、文档列表、问答测试 |
| 爬虫 | 本地 Mac 跑（Playwright） | 不租 VPS；定时同步；HTTP 推回 Cloudflare |
| 域名 | **不买** | 用 `unequal.xxx.workers.dev` 即可 |

## 四、需求细化

### 4.1 用户角色

| 角色 | 入口 | 能力 |
|---|---|---|
| 家长（终端用户） | 微信小程序 | 提问、查看答案、点开引用原文 |
| 管理员（你） | Web 管理后台 | 增删源、查看文档、问答测试、查看抓取日志 |

### 4.2 数据来源优先级

| 优先级 | 来源 | 实现难度 | 备注 |
|---|---|---|---|
| P0 | 文件上传（PDF / Word / TXT / MD） | 低 | MVP 必须 |
| P1 | 指定网页 URL | 中 | Cloudflare Browser Rendering 或本地 Playwright |
| P2 | 小红书指定账号 | 高 | 反爬严格，初期人工导出 + JSON 导入 |
| P2 | 微信公众号指定账号 | 高 | 搜狗微信或第三方聚合 |

### 4.3 引用追溯机制（核心需求）

每条回答必须满足：

- LLM prompt 强制"只基于检索结果回答"
- 引用编号 `[来源 N]` 从检索结果的 chunk 列表里选，不允许自由发挥
- 前端把 `[来源 N]` 渲染成可点击卡片：标题 / 摘要 / 原文链接
- chunk 元数据必须包含：`source_id` / `doc_id` / `chunk_id` / `title` / `url` / `snippet`

## 七、关键设计要点

### 7.1 数据模型

```sql
-- D1 schema
CREATE TABLE source (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,          -- 'file' | 'webpage' | 'xiaohongshu' | 'wechat-mp'
  title TEXT,
  url TEXT,
  account TEXT,                -- 账号 / 作者
  created_at INTEGER,
  meta TEXT                    -- JSON 元数据
);

CREATE TABLE document (
  id TEXT PRIMARY KEY,
  source_id TEXT NOT NULL,
  title TEXT,
  raw_path TEXT,               -- R2 路径（PDF / 原文）
  parsed_text TEXT,            -- 解析后纯文本
  created_at INTEGER,
  FOREIGN KEY (source_id) REFERENCES source(id)
);

CREATE TABLE chunk (
  id TEXT PRIMARY KEY,
  document_id TEXT NOT NULL,
  source_id TEXT NOT NULL,
  idx INTEGER,                 -- 在文档内的顺序
  content TEXT,
  token_count INTEGER,
  FOREIGN KEY (document_id) REFERENCES document(id)
);

-- Vectorize 中按 chunk_id 索引，metadata 镜像 chunk 表
```

### 7.2 检索 + 生成 pipeline

```
1. 用户问句 → MiniMax embedding → 向量 q
2. q → Vectorize top-K(20) 召回
3. 重排 / 截断 → top-K(5)
4. 把 5 个 chunk 拼成 context，附引用编号 [1]..[5]
5. MiniMax chat：system prompt + context + 用户问题 → 回答
6. 回答里出现的 [来源 N] → 前端映射到引用卡片
```

### 7.3 反幻觉机制

- system prompt 明确禁止"超出 context 编造"
- 强制从给定 chunk 编号里挑引用
- 答案里出现的引用编号必须能在返回的 chunk 列表里找到对应
- 找不到时降级为"未在知识库中找到可靠来源"
