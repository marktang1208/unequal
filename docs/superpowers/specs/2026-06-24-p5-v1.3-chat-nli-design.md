## 1. 目标与非目标

### 1.1 目标

- /api-chat 后置插入 NLI 蕴含验证,复用 v1.1.1 `HttpNliProvider`(硅基流动 Qwen2.5-7B)
- 复用 v1.2 降级触发条件:`cleaned.length < NLI_MIN_ANSWER_LEN (100)` 跳过 NLI
- verdict !== "entailed" → 给 answer 加 `⚠️ 以下回答部分参考资料未提及/存在冲突` prefix
- 仅 reject 时写 audit,`action = "chat_nli_reject"`(区别于 ask 的 `ask_nli_reject`)
- 失败降级:runtime/timeout → NoopNliProvider 风格 verdict (entailed),不阻塞 chat,fail-open
- 持久化 `messages` 时存 `finalAnswer`(用户看到的就是持久化的)
- 真接 5 步验证 PASS,生产 0 新 env var(20 vars 已就绪)

### 1.2 非目标

- 不改 LLM prompt / 不重生成(同 v1.1)
- 不切片 NLI 推理(同 v1.1)
- 不存 chat 历史消息全文到 NLI premise(隐私/PII)— 只用**当前轮** cleaned answer
- 不扩 multi-user session 维度
- v2 候选:跨轮 NLI、history 摘要 NLI、流式 NLI、跨 chat session 趋势分析

## 2. 架构与数据流

### 2.1 改动位置

`apps/api/src/handlers/api-chat.ts:226-284`(LLM chat 完成 → 持久化 → 返回前)

### 2.2 核心流程

```
1-7. 现有 chat 流程(L254-274,LLM chat + 持久化 + [N] 解析)
    - LLM 答案:answer = result.content || "(无回答)"
    - 持久化:newMessages = [...session.messages, user, assistant(answer)]
    - 解析:parseAnswerSegments(answer, topChunks.length) → citedNums

8. P5 v1.3 NLI 后置插入(NEW):
   8a. cleaned = parseAnswerSegments(answer, topChunks.length).cleaned
   8b. nliMinLen = getNliMinAnswerLen()
   8c. if shouldSkipNli(cleaned, nliMinLen):
         console.log(`[nli] skipped: answer too short (${cleaned.length} < ${nliMinLen})`)
         // 跳过 NLI,但继续走 session 持久化(用 raw answer),不调 NLI,不写 audit
         await persistSessionMessages(session, q, answer, isNewSession, now, title)
         return jsonResponse({ answer, citedNums, citations, session_id, session_title, is_new_session: isNewSession })
   8d. nliHypothesis = topChunks.map(c => c.content).filter(l>0).join("\n\n")
   8e. try: verdict = await provider.verify(cleaned, nliHypothesis); recordNliSuccess()
       catch(err): recordNliFailure(err); verdict = { entailed, score=1, ... } (fail-open)
   8f. 仅 reject 时写 audit(action = "chat_nli_reject",actor.sessionId = session.id)
   8g. finalAnswer = nliSucceeded ? applyWarning(answer, verdict) : answer
   8h. 持久化 messages 时 assistant.content = finalAnswer(覆盖原 answer)

9. 返回 jsonResponse({ answer: finalAnswer, citedNums, citations, session_id, session_title, is_new_session })
```

**修订说明**:8c 跳过路径仍走 session 持久化(用 raw answer),避免短问题用户刷新页面丢历史。详见 §6。

### 2.3 关键决策

- **入参 premise**:用 `cleaned`(去掉 `[N]` 标记)— 同 ask
- **入参 hypothesis**:top-5 chunk 内容拼接 — 同 ask
- **持久化**:`finalAnswer`(带 ⚠️ prefix)— 用户回看 chat 历史看到的就是带 warning 的版本
- **session_id 写入 audit**:`actor.sessionId = session.id`(新字段,与 ask 区分)
- **actor.via**:`"jwt"`(区别于 ask 的 `"admin_token"`)

## 3. 数据流 / 错误处理 / 边界

### 3.1 错误处理矩阵

| 场景 | 行为 | audit 写? | 用户看到 | 持久化 |
|---|---|---|---|---|
| cleaned 长度 < NLI_MIN_ANSWER_LEN (100) | skip,直接 return(同 v1.2 ask 跳过路径) | ❌ 不写 | 原 answer | 原 answer |
| NLI 成功 + verdict = entailed | pass,继续走持久化 | ❌ 不写 | 原 answer | 原 answer |
| NLI 成功 + verdict ∈ {neutral, contradiction} | applyWarning 加 ⚠️ prefix | ✅ `chat_nli_reject`, `result: "success"` | 带 ⚠️ 的 answer | 带 ⚠️ 的 answer (finalAnswer) |
| NLI runtime 错误 | fail-open,verdict=entailed | ✅ `chat_nli_reject`, `result: "failure"`, `error: "nli_runtime_error"` | 原 answer | 原 answer |
| NLI timeout (> 15s) | fail-open,verdict=entailed | ✅ `chat_nli_reject`, `result: "failure"`, `error: "nli_timeout"` | 原 answer | 原 answer |
| 进程内 5min cache / 10-timeout 永久降级 | 复用 v1.1 get-provider 单例 | ❌(同 ask,内部状态) | 原 answer | 原 answer |

### 3.2 持久化边界

- 持久化用 `finalAnswer`(L228-231 `newMessages` 中 assistant.content)
- L256 `parseAnswerSegments(answer, ...)` 仍用 raw `answer` 解析 `[N]`(保持 [N] 引用未污染)
- L268 `citedNums`/`citations` 基于 raw answer 解析,与 ask 一致
- 跳过路径(8c)→ 用 raw answer 持久化,与正常路径行为一致

### 3.3 PII 防护

- audit 只存 `queryHash = sha256(q).slice(0,16)` — **不存** chat 历史消息、session 标题
- `actor.sessionId = session.id`(ULID,非 PII)
- `request.title` = `q.slice(0, 100)`(同 ask)— chat q 通常短问句,可接受
- `target.userId` = JWT payload.sub — 用户 PII 已存,无新增

### 3.4 chat 特有的兜底

- LLM 答案兜底:`answer = result.content || "(无回答)"` — skip path 同 v1.2
- LLM 失败(chat L221-224): `MINIMAX_FAILED` 502,**NLI 不调用**(上游已经返 5xx)
- JWT 失败(L98-99): `AUTH_FAILED` 401,**NLI 不调用**
- chunks 为空:继续走 NLI(nliHypothesis = "",premise = cleaned)— v1.1 handle 这种情况返回 verdict=neutral(无证据支持)

## 6. §2.2 修订说明(自审发现)

**原文**:8c 早 return 路径上,跳过 NLI 调用,所以**不持久化** session messages。这意味着短问题 → 短答案 → skip → 新 session 不持久化。

**问题**:chat 场景下,即便 skip NLI,用户期望"我问了一句话 + 系统给了回答"被记录到历史。如果不持久化,用户刷新页面或退出重登会丢失对话。

**修订**:
```
8c. if shouldSkipNli(cleaned, nliMinLen):
       console.log(`[nli] skipped: answer too short (${cleaned.length} < ${nliMinLen})`)
       // 跳过 NLI,但**继续**走 session 持久化(用 raw answer),不调 NLI,不写 audit
       // 然后 return
```

实现层面:把 session 持久化代码(L227-251)抽取到一个独立分支,在 skip path 也调用,但不调 NLI、不写 audit。

**修订后流程**:
```
1-7. 现有 chat 流程(LLM chat + parseAnswerSegments)
8a. cleaned = parseAnswerSegments(answer, topChunks.length).cleaned
8b. nliMinLen = getNliMinAnswerLen()
8c. if shouldSkipNli(cleaned, nliMinLen):
       console.log(`[nli] skipped...`)
       await persistSessionMessages(session, q, answer, isNewSession, now, title)
       return jsonResponse({ answer, citedNums, citations, session_id, session_title, is_new_session: isNewSession })
8d. nliHypothesis = ...
8e-h. (正常 NLI 路径 + finalAnswer + 持久化 finalAnswer)
9. return jsonResponse({ answer: finalAnswer, ... })
```

这样 skip path 也保证 session 持久化,用户刷新不丢历史。

## 7. 边界

- **HttpNliProvider only**(云端推理)— 同 v1.1
- **短答案 < 100 字符 skip**— 同 v1.2,默认 `NLI_MIN_ANSWER_LEN=100`
- **仅 reject 写 audit**(pass 不写减少噪声)— 同 v1.1
- **失败 audit**(runtime_error / timeout)— 同 v1.1
- **fail-open**(网络故障不阻塞 chat)— 同 v1.1
- **持久化 finalAnswer**(用户回看一致)— 新增
- **actor.via = "jwt"** + **actor.sessionId = session.id** — 与 ask 区分
- **不动 prompt / 不切片 NLI / 不重生成** — 同 v1.1
- **仅当前轮 answer 作 premise**(不存 chat 历史)— 隐私/PII 防护
